# Backlog — Action Items from Interview-Prep Reviews

Granular, prioritized queue of concrete fixes/features decided while working through [[MOC - Interview Prep Baseline]] via the `/interview-review` command. Distinct from the phase-level roadmap in `docs/playbook-strategy.md` §6 (Phase 1-4, strategic/product sequencing) — this is the tactical "what do we act on, in what order" list.

Order within each section is priority order: **top = do next.** New items are inserted at the position matching their priority, not appended to the bottom. Bugs outrank features of similar importance.

## Bugs

- 🔴 **[Bug]** Deleting a category or content record that still has relations (parent/child edges, category assignments) should not silently cascade — raise a confirmation-style error surfacing what it's connected to ("related to N parent(s) / M child(ren) — delete anyway?") so a manager can re-parent/re-assign first if needed. Applies once delete endpoints for content records/categories are built (none exist yet); pairs with the FK/cascade design from Content Q5. Does not cover the harder historical-data question below — see the 📋 Needs ADR item. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q6.

- 🔴 **[Bug]** `email_modules/registry.py`'s `_REGISTRY` is built once at import time and never refreshed — template *HTML* re-reads fresh from disk on every render, but the *manifest* (variables, cms flag, label) stays frozen until restart, an inconsistent freshness split within the same registry. No "save template" endpoint exists to hook a notification into anyway — templates get added by dropping files directly into `storage/email_modules/`, outside any request lifecycle. Fix: real hot-reload instead of restart+notify — `_discover()` is already side-effect-free, so re-run it (behind a cheap directory-mtime check, or via an explicit manual "reload registry" admin action) instead of only running it once at import. Same stale-until-restart pattern exists in `decision/strategies/registry.py` too — worth applying consistently, though not expanding scope here beyond what was asked. Source: [[Interview Prep - Snapshots, Rendering, Email Modules]] → Email Modules Q2.

- 🔴 **[Bug]** `ingest_provider_event` (`providers/service.py:15-27`) raises `ValueError` → HTTP 400 and the event is simply lost when no matching `DeliveryExecutionDB` is found via `provider_message_id` — no quarantine/retry mechanism exists, directly violating [[ADR-129 — Correlate Provider Events to Delivery Executions]]'s "must not be silently discarded" requirement. Fix: store unmatched events (full payload) in a quarantine/dead-letter table for later reconciliation instead of returning an error and losing the data. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Providers Q2.

- 🔴 **[Bug]** `_load_brand_css` (`rendering/service.py`) re-reads and re-inlines `brand.css` from disk on every single render with no caching — becomes O(recipients) redundant work if `render_variant_html` is called per-recipient (already planned, see the Delivery Q2 personalization fix). Fix: cache the CSS text (e.g. `functools.lru_cache` or module-level cache). Cheap, low-risk fix regardless of the pending MJML migration (Email Modules Q1) — raw-HTML templates keep needing Python-side inlining even after MJML lands, since ADR-131 keeps raw HTML as a permanent escape hatch, not just a stopgap; this just applies to fewer templates long-term. Source: [[Interview Prep - Snapshots, Rendering, Email Modules]] → Rendering Q2.

- 🔴 **[Bug]** `create_snapshot_for_variant` (`snapshots/service.py:96-138`) derives its HTML and its `render_context` metadata from two independent, sequential `DecisionResolutionDB` queries (inside `render_variant_html` and `build_render_context` respectively) instead of one shared resolution pass — if a new decision resolution is inserted for the same slot+recipient in the narrow window between the two calls, the snapshot's metadata can describe different content than what's actually baked into the HTML file, undermining ADR-062's audit-trust guarantee. Fix: derive both from one resolution pass — have `render_variant_html` return/expose the resolution details it used, and have `build_render_context` reuse them instead of re-querying. Source: [[Interview Prep - Snapshots, Rendering, Email Modules]] → Snapshots Q3.

- 🔴 **[Bug]** `RecipientDB.preferred_airport` is a domain-specific example (the airline use case) hardcoded as a first-class column in the generalized reference schema — wrong for a reference architecture meant to be reused across any industry. The data itself can stay airline-related as a living example, but the schema shouldn't hardcode it. Fix: remove `preferred_airport` (and any other column tied to one specific use case) and represent business-specific fields through a generalized key/value or JSON structure instead (a "preferences" JSON, or extend `attributes`) — same direction as the `attributes` scoping fix below. `language` stays as-is (a genuinely universal field, not domain-specific). Source: [[Interview Prep - Delivery, Providers, Recipients]] → Recipients Q6 (addendum, flagged for follow-up — don't forget).

- 🔴 **[Bug]** `RecipientDB.attributes` is a completely open JSON blob with no documentation or validation bounding what belongs in it — nothing stops it drifting into CRM-scope data (service cases, addresses, invoices) that [[ADR-126 — Maintain Local Recipient Projection]] explicitly forbids ("must not become... a full customer profile repository... a system of record for customer data"). Confirmed: a rich profile growing over time (especially via AI/decisioning) is fine and expected — the actual boundary to enforce is engagement/personalization-relevant data only, never CRM-owned data types. Fix: document and validate the allowed shape/keys for `attributes` so it can't silently absorb forbidden data. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Recipients Q6.

- 🔴 **[Bug]** `create_recipient` has no duplicate check before insert — a repeat CRM sync (a realistic, expected operational path) hits an unhandled `IntegrityError` on `external_id`'s unique constraint and returns a raw 500 instead of a clean upsert. Fix: upsert keyed on `external_id` (update if exists, insert if not) instead of blind insert. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Recipients Q5.

- 🔴 **[Bug]** `RecipientPreferenceDB` has no unique constraint on `(recipient_id, category_id)` despite being meant as the current/aggregate running-total score per pair (distinct from `PreferenceUpdateLogDB`, which is the correct append-only historical log and needs no such constraint). `create_recipient_preference` always inserts rather than upserting, so duplicate "current score" rows can silently appear, and `apply_event_to_preferences`'s `.first()` picks one non-deterministically, leaving the other permanently stale. Fix: add a unique constraint on `(recipient_id, category_id)` and change creation to upsert (insert if missing, update in place if it exists). Source: [[Interview Prep - Delivery, Providers, Recipients]] → Recipients Q3.

- 🔴 **[Bug]** `RecipientDB.email` has no unique constraint, so duplicate-email rows are possible (e.g. from a bad import) and nothing catches it — whether email *should* be hard-unique is a business decision (deferred, likely part of the future config-layer work), not decided now. Regardless of that decision, small fix needed now: deduplicate by email before sending and when calculating audience segments/criteria, so duplicate-email recipients don't get double-emailed or get double-counted in segment previews. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Recipients Q2.

- 🔴 **[Bug]** No uniqueness check on `provider_event_id` at ingestion — a webhook delivered twice creates a second `EngagementEventDB` row unconditionally (`providers/service.py`). This is a different, earlier layer than the Insight Q3 dedup fix (which governs whether a *scoring* update applies, not whether the raw event itself gets recorded twice). Fix: add a uniqueness check/constraint on `provider_event_id` at ingestion. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Providers Q5.

- 🔴 **[Bug]** `DeliveryExecutionDB.provider_message_id` has no unique constraint or index (`delivery/db_models.py:32`), despite being the sole correlation key for every inbound provider webhook per ADR-129 — collisions would silently pick the wrong execution, and the lookup (the hot path for all inbound events) degrades linearly with volume. Fix: add a unique constraint + index. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Providers Q3.

- 🔴 **[Bug]** `send_send_instance` (`delivery/service.py:160-173`) only calls `db.commit()` once, at the very end of the send loop — if `provider.send()` raises mid-batch (e.g. execution 5 of 100), executions 1-4 were actually sent by the provider but never get their `"sent"` status persisted, since the exception aborts the function before the single end-of-loop commit. Keeping "stop the batch on a failure" as the safe default is correct (provider-dependent whether skip-and-continue would even be safe) — but whatever succeeded before the stop must still be recorded accurately. Fix: commit incrementally (or otherwise ensure completed executions are persisted) so the DB never claims an email was never sent when it actually was, independent of retry/skip policy. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Delivery Q3.

- 🔴 **[Bug]** `send_send_instance` never consults `result.success` from `provider.send()` — `execution.status` is unconditionally set to `"sent"` regardless of the actual result (`delivery/service.py:168`), and the mock provider always returns `success=True` so this path is never exercised. Distinct from Q3 (that's an exception being raised; this is a returned failure result being ignored). Fix: consult `result.success` when setting status, and log the full result (success/failure/message) for every send attempt, not just failures — small observability fix, useful regardless of provider. Defer the harder part (how to react differently to varied real-provider failure modes/retry semantics) until real providers are integrated — building that against a mock that only ever succeeds would be guesswork. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Delivery Q6.

- 🔴 **[Bug]** Nothing stops two concurrent calls from sending the same send instance twice — no status guard before the loop runs in `send_send_instance`, no unique constraint or row lock (`delivery/service.py`). Would produce duplicate sends and last-write-wins on `provider_message_id`. Fix: add a status guard (reject/no-op if already `"sending"`/`"sent"`) plus a row lock or unique constraint to make it safe-by-construction, not just application-level. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Delivery Q5.

- 🔴 **[Bug]** `POST /decision/slots/{id}/execute` (`decision/router.py:24-37` → `decision/service.py:9-43`) unconditionally inserts a brand-new `DecisionResolutionDB` row every time it's called, with no check against the recipient's last resolution for that slot — confirmed to already happen in practice (re-running for a recurring journey creates another row even when nothing changed). Two fixes, prioritized data-minimization first: (1) non-personalized strategies (`top_score`, `requires_recipient=False`) should only ever have **one** resolution row per slot (`recipient_id=NULL`, already supported by the schema and rendering's read path) — never a duplicate per recipient; (2) personalized strategies (`recipient_top_score`) should only write a new row when something that could change the outcome actually changed since the recipient's last resolution (preference update, candidate content change) — "no new signal, keep the last recommendation" — rather than accumulating an unbounded, ever-growing number of per-person records over time. Storing as little per-recipient data as possible matters more here than the query-performance win, though both improve together. Source: [[Interview Prep - Decision, Overrides, Insight]] → Decision Q5.

- 🔴 **[Bug]** `send_send_instance` reads one shared, variant-level snapshot and sends identical HTML to every recipient (`delivery/service.py:111-172`) instead of resolving per-recipient HTML — even though decision-slot personalization (ADR-083) already exists in the rendering layer and could resolve different content per recipient within the same variant. Note: providers genuinely don't need campaignId/variantId/snapshotId at all (internal bookkeeping, not the ESP's concern), and variantId being identical across recipients within one send is *correct*, not a gap, when personalization stays inside one variant via decision slots — it would only legitimately differ per recipient in a true split-delivery/A/B scenario, which is the separate "shadow variant" idea already in Needs ADR. Fix scope, narrowed: resolve/render HTML per recipient (reusing existing per-recipient rendering) before handing each recipient their own final HTML to the provider, instead of reusing one shared snapshot for the whole send. Subject/preheader as dedicated provider fields is tracked separately (Campaigns Q4). Source: [[Interview Prep - Delivery, Providers, Recipients]] → Delivery Q2.

- 🔴 **[Bug]** `RecipientTopScoreStrategy`'s tie handling (`recipient_top_score.py:69-76`) is undisclosed, unintentional pseudo-randomness — `max()` picks whichever tied candidate the DB happened to return first, with no `ORDER BY` and no mention of the tie in `reason`, undermining ADR-085 explainability and risking a permanent, unexplainable bias toward whichever content has a lower ID. Fix: make tie-breaking an explicit, disclosed random selection among tied candidates, and state it in the `reason` string ("tied at score X between N candidates, selected randomly") — honest and auditable rather than hidden. Source: [[Interview Prep - Decision, Overrides, Insight]] → Decision Q7.

- 🔴 **[Bug]** Both plugin-style registries (`decision/strategies/registry.py:11-29`, `email_modules/registry.py:42-64`) eagerly discover and load every file at import time with no per-item error isolation — one broken strategy file (bad import, failing constructor) or one malformed `.json` manifest crashes the entire registry, taking down the whole app at startup. `email_modules/registry.py` already isolates the *missing-counterpart-file* case gracefully (logs a warning, skips just that pair) but not malformed content within a file. Fix: wrap each individual file's discovery/import/instantiation in its own try/except inside both `_discover()` loops — log a clear warning naming the broken file, skip just that one, keep the rest of the registry working. Auto-discovery ("drop a file, no code changes") and full graceful degradation aren't in tension — this is a standard pattern, not a tradeoff. Apply to both registries in one pass. Source: [[Interview Prep - Decision, Overrides, Insight]] → Decision Q1 (also affects the not-yet-reviewed email-modules question on the same gap).

- 🔴 **[Bug]** Two competing category hierarchy mechanisms exist: `CategoryDB.parent_category_id` (leftover from an earlier, wrong implementation) and `CategoryRelationDB` (parent_category_id/child_category_id/relation_type edge table, already what the graph view at `frontend/router.py:1608-1621` actually uses). Fix: drop `CategoryDB.parent_category_id` entirely and consolidate on `CategoryRelationDB` as the single mechanism. Deliberately keep it fully flexible, no new restrictions: a child may have multiple parent relations, and parent-level (main) categories may relate to each other or to children freely — needed for real-world multi-parent taxonomies and for AI-discovered relations later, and adding artificial constraints now isn't necessary. Do add a cycle guard (reject a relation that would create a direct or transitive parent_child loop, e.g. A→B and B→A) since that's a genuine logical contradiction, not a business-flexibility case. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q5.

- 🔴 **[Bug]** `DecisionResolutionDB.score` is stored as `Integer`, so `score=int(result.score)` (`decision/service.py:42`) silently truncates the weighted float score `recipient_top_score` computes — two candidates differing only in fractional score (12.4 vs 12.9) both round to 12, losing tie-breaking/explainability granularity relevant to ADR-085. Fix: change the column to a proper decimal/float type (e.g. `Numeric`) and drop the `int()` cast. Source: [[Interview Prep - Decision, Overrides, Insight]] → Decision Q6.

- 🔴 **[Bug]** `ModuleInstanceDB` has both `content_record_id` and `decision_slot_id` as independent nullable FKs with nothing enforcing mutual exclusivity — if both are set, rendering silently prefers `content_record_id` and ignores the decision slot (`rendering/service.py:196-205`); if neither is set, the module renders nothing. Considered replacing with a generic `type`+`id` polymorphic column but rejected — that's a known anti-pattern that loses real FK referential integrity and forces conditional/manual joins, for a type-count (2) too small to justify it. Fix: keep the two nullable FK columns, add a mutual-exclusivity constraint (DB `CHECK` or Pydantic/service-layer validator) enforcing exactly one is set. `campaigns/db_models.py:37-53`, `models.py:50-55`. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q2.

- 🔴 **[Bug]** `recipient_id` means two different things across modules — `recipients`/`decision`/`insight` use the internal integer PK, while `delivery`/`providers` (`DeliveryExecutionDB.recipient_id`) use the string `external_id`, with no FK between them. This forces indirect lookups (e.g. `insight/service.py:105-118` has to query `RecipientDB WHERE external_id = ...` just to translate a delivery execution's recipient into something joinable), and each such lookup is an extra failure mode (mismatched/stale external_id → `ValueError`) that a real FK wouldn't have. Confirmed this doesn't add any new CRM integration burden — per [[ADR-126 — Maintain Local Recipient Projection]], delivery execution tracking already requires the local `RecipientDB` projection to exist and be synced regardless of which of its two ID columns gets referenced; switching to the internal PK is purely internal plumbing, not a new sync dependency. Fix: make `DeliveryExecutionDB.recipient_id` (and any other cross-module reference doing the same thing) a direct FK to `RecipientDB.id`, consistent with [[ADR-054 — Use Internal Recipient Identifiers]]. Good time to do this — data model is still cheap to reshape. Source: [[Interview Prep - Decision, Overrides, Insight]] → Insight Q5 (also flagged as a cross-cutting finding in [[MOC - Interview Prep Baseline]], affects the not-yet-reviewed Delivery/Recipients cluster too).

- 🔴 **[Bug]** `ContentCategoryAssignmentDB.score` has no validation at write or read time — can be zero (degenerate/meaningless; "no relevance" should mean no assignment row, not a stored zero) or exceed the 0-10 range, and `RecipientPreferenceDB.new_score` has no ceiling so it can grow unbounded, compounding into `RecipientTopScoreStrategy`'s ranking (`recipients/service.py:135,170`). Fix now: reject a score of exactly zero at write time. The 0-10 range itself is a POC-only convention, not a hard rule — fold into the existing config-layer future work (Insight Q2) so a deployment can set its own limits rather than hardcoding 0-10. Source: [[Interview Prep - Decision, Overrides, Insight]] → Insight Q4.

- 🔴 **[Bug]** The dedupe key in `apply_event_to_preferences` — `(recipient_id, category_id, reason=event_type, delivery_execution_id)` — doesn't identify the specific event, so two distinct legitimate engagements (e.g. clicks on two different links, same email/category/execution) get conflated: the second is silently dropped as a "duplicate" and never scored (`recipients/service.py:137-150`). Every real event must still be saved (already true for the raw `EngagementEventDB` row) — deduplication should only decide *whether/how much score* a given event contributes, not whether the event itself is recorded. Fix: include something that uniquely identifies the specific engagement (e.g. `event_id`) in the dedupe key, so distinct events each get their own scoring decision while true duplicate deliveries of the *same* event still dedupe correctly. Source: [[Interview Prep - Decision, Overrides, Insight]] → Insight Q3.

- 🔴 **[Bug]** `OverrideEventDB.override_type` is unconstrained free text — only `"content" | "segment"` documented in a code comment, not validated anywhere. Fix: make it a proper enum. Small, standalone fix, separate from the reset-mechanism question logged under the override-strategy redesign Feature. Source: [[Interview Prep - Decision, Overrides, Insight]] → Overrides Q5.

- 🔴 **[Bug]** `record_outcome_delta` (`overrides/service.py:32-39`) replaces `outcome_delta` wholesale on each PATCH instead of merging keys — outcome data that arrives incrementally (e.g. open-rate today, click-rate a week later) has the second update silently overwrite the first entirely. `test_overrides.py:130-138` only checks non-`outcome_delta` fields survive; the merge case is untested and would currently fail. Fix: merge incoming keys into the existing `outcome_delta` dict instead of replacing it. Source: [[Interview Prep - Decision, Overrides, Insight]] → Overrides Q3.

- 🔴 **[Bug]** `record_outcome_delta` (`overrides/service.py:32-39`) is a read-modify-write with no lock — concurrent PATCH calls computing outcome deltas for the same `override_id` (e.g. an open-rate job and a click-rate job overlapping) have last-write-wins semantics, silently dropping whichever update committed first. Compounds the Q3 merge bug above. Fix mechanism not yet decided — either optimistic locking (a `version`/`updated_at` check-and-increment on write) or a DB-level row lock (`SELECT ... FOR UPDATE`); needs a decision before implementing. Source: [[Interview Prep - Decision, Overrides, Insight]] → Overrides Q4.

- 🔴 **[Bug]** `OverrideEventCreate` (`overrides/models.py:6-14`) has no validation that `system_content_record_id` and `override_content_record_id` actually exist or differ from each other — FK enforcement depends on whether `PRAGMA foreign_keys` happens to be on for the dev DB, so a bad ID silently creates an orphaned audit row. Same systemic silent-orphan pattern as Campaigns Q5 below. Fix: validate both referenced IDs exist (and aren't identical) before insert. Source: [[Interview Prep - Decision, Overrides, Insight]] → Overrides Q2.

- 🔴 **[Bug]** `create_decision_resolution` (`campaigns/service.py:235-257`) accepts arbitrary `content_record_id`/`content_version_id`/`recipient_id` with no existence validation before insert — depending on whether the DB engine enforces FK constraints (often off by default in SQLite), a bad ID silently creates an orphaned resolution row that only breaks later, at rendering's join-based lookup. Fix: no orphan row should ever be silently accepted — validate referenced IDs exist before insert (or ensure FK enforcement is actually on, e.g. `PRAGMA foreign_keys=ON` for SQLite) and return a clear error otherwise. Worth checking whether other `create_*` functions across the codebase have the same silent-orphan gap once this is fixed, since it's likely a systemic pattern, not unique to this one function. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q5.

- 🔴 **[Bug]** `add_member`/`bulk_add_members` (`audience/service.py:54-69,72-85,125-134`) do a SELECT-then-INSERT with no unique constraint on `(group_id, recipient_id)` — a TOCTOU race under concurrent calls can produce duplicate membership rows. Fix: add a DB-level unique constraint on `(group_id, recipient_id)` on `AudienceGroupMemberDB`, making it safe-by-construction instead of relying on the application-level existence check. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q3.

- 🔴 **[Bug]** `AudienceGroupDB.name` uniqueness is case-sensitive and unhandled — "Newsletter VIPs" and "newsletter vips" succeed as two distinct groups, and a genuine duplicate name raises an unhandled 500 instead of a clean error (`router.py:16-18`, `service.py:15-33`). Fix: enforce case-insensitive uniqueness and catch `IntegrityError` to return a clean 409 with a clear message. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q5.

- 🔴 **[Bug]** `create_campaign` (`campaigns/service.py:33-61`) auto-creates the initial `VariantDB` via two independent `db.commit()` calls rather than one transaction — a failure between them (e.g. DB connection drop) leaves a persisted campaign with zero variants, violating the "a campaign always has a variant" invariant. Fix: wrap both inserts in a single transaction (flush + one commit at the end) so it's atomic. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q1.

- 🔴 **[Bug]** `create_snapshot_for_variant` builds a throwaway `render_context` dict that's immediately overwritten by `build_render_context`'s real result — dead code, no behavioral impact. Fix: remove the discarded first assignment. Trivial cleanup, lowest priority. Source: [[Interview Prep - Snapshots, Rendering, Email Modules]] → Snapshots Q1.

## Features

- 🔴 **[Feature]** Minimal consent gate: add a `consent_status` field to `RecipientDB` (opted-in/opted-out/pending, synced from CRM — already permitted by ADR-126's "communication preferences" as an allowed Recipient Projection field), plus a consent-sync log to detect CRM↔platform drift. Filter non-consenting recipients at **audience-resolution time**, before decision/rendering run at all, not as a late gate before the provider call — driven by both a legal reason (GDPR processing scope, needs real legal review) and a cost reason (avoid paying for AI/token-driven decisioning on recipients who'll never receive anything). Closes a real gap providers can't be trusted to close themselves (some, e.g. Selligent, ignore opt-outs on API-triggered sends). Ranked high — resolves the previously-deferred, highest-priority item from the technical review's Recipients Q1. Source: `docs/business-interview-baseline.md` §F1 (via `/business-review`).

- 🔴 **[Feature]** New idea, not yet designed: cost/token feedback before running an AI-driven decision batch — as decisioning increasingly routes through paid AI models or automation platforms (e.g. n8n), surface expected cost/token usage before committing to a run, for the recipients who *will* receive something (separate from the consent gate above, which is about not running decisioning for people who won't). Ties into the pluggable AI-capability extension points from the E1 resolution and the not-yet-built Phase 3C/3D AI work. Source: `docs/business-interview-baseline.md` §M3 (via `/business-review`).

- 🔴 **[Feature]** Small defensive safety-net check: refuse/alert if a fully rendered email ends up with literally zero content modules before sending — purely to catch accidental misconfiguration (e.g. every module in a variant is decision-slot-driven and all happened to resolve to nothing), not a business-policy decision. Confirmed the composition model (static content blocks) and strategy-plugin system already let a company guarantee content by construction if they want to — this is just a cheap guard against the accidental case, lowest priority. Source: `docs/business-interview-baseline.md` §E3 (via `/business-review`).

- 🔴 **[Feature]** **Required before public launch** (confirmed via `/business-review`, `docs/business-interview-baseline.md` §H1) — a real provider integration proves the "no vendor lock-in" pitch is real, not just a design promise; deliberately sequenced late (after remaining frontend work) since it costs real money. Also depends on this adapter layer existing first. Complete the inbound provider adapter layer (`providers/adapters/`, currently an empty mock file) — build the actual translation contract (e.g. an abstract base analogous to `DeliveryProvider` for outbound sends) that turns a raw provider webhook payload into the canonical `ProviderEventCreate` shape, since `ingest_provider_event` today just accepts an already-normalized payload directly with no real adapter behind it. Fits the "architecture + one worked example" strategy (per the provider-plugin-strategy decision in project memory) — this is the generic interface/contract any future custom connector would implement, not a vendor-specific plugin, so it doesn't conflict with the earlier decision to not ship maintained ESP/CRM plugins. **Must also own event-type normalization**: `event_type` is currently unconstrained free text with only a hardcoded `"click"` check giving it any meaning downstream (`insight/service.py`'s `EVENT_PREFERENCE_DELTAS`) — bounce/complaint (mandatory per ADR-106) are accepted into storage but never actually handled. Validation/normalization must be centralized in the adapter layer, not a single hardcoded enum, since the raw event-type vocabulary differs per provider (e.g. one ESP's `"hard_bounce"` and another's `"permanent_failure"` both need to normalize to the same canonical internal type). **Concrete reaction to bounce/complaint, now that the consent gate is resolved (see the consent-gate Feature above, from `docs/business-interview-baseline.md` §F1):** a bounce/complaint event should (1) update the recipient's `consent_status` (revoke/mark opted-out) and relay back to the CRM if there's no direct CRM↔provider webhook connection, since this platform is the only place positioned to notice; and (2) separately feed the Insight layer as a content-quality signal ("this content triggered a lot of complaints") — a distinct, analytics-flavored reaction to the same event, not the same action as the consent update. **Design approach, when this work starts:** favor (b) "fairly simple to adjust to your provider" over (a) "one contract that auto-fits every ESP generically" — real webhook payloads have enough genuine quirks (auth/signature schemes, batch-vs-single-event delivery, inconsistent field naming) that a fully generic auto-detecting layer would be over-engineering and fragile. Same pattern as the outbound side: a small, clean canonical contract (a `ProviderAdapter` interface mirroring `DeliveryProvider`) plus one well-documented worked example. Before finalizing the canonical event shape, pull real webhook API docs from a small, deliberately varied set of providers (not guess from memory, which may be stale) — suggested: one already referenced in the ADRs (Brevo or Resend), one classic/well-documented one (Mailgun or Postmark), and SES given it's common infrastructure for the target buyer. Enough variety to find genuine common ground without over-researching providers that'll never be used. Source: [[Interview Prep - Delivery, Providers, Recipients]] → Providers Q1, Q4; `docs/business-interview-baseline.md` §F2, §H1 (via `/business-review`).

- 🔴 **[Feature]** Systemic gap: the project needs a general pattern for concurrency guards and validation checks across write endpoints, not just ad hoc per-issue fixes. ## Needs ADR (blocked on design decision)

_Not prioritized alongside 🔴 To Do items — these need a decision recorded as an ADR before they're actionable. Move to Bugs/Features once resolved._

- 📋 **Finalize the rendered-HTML/snapshot storage strategy — now more urgent given per-recipient personalization.** Currently writes flat HTML files (`storage/snapshots/`), with `SnapshotDB.html_storage_type` as an unused extension point for object storage (per project notes: this was already an open decision). Commit 380ebe1 worked around a related seed-data bug (bad `html_location` paths) rather than resolving the underlying strategy, and left a side-effect NULL in `PreferenceUpdateLogDB.event_id` (a `nullable=False` column). The Delivery Q2 fix (per-recipient HTML resolution instead of one shared variant-level snapshot) raises the stakes: instead of one rendered artifact per variant, this could mean one per recipient per send — making "do we even need to persist full HTML files at all" a much more pressing question than before. Needs a decision on: whether to store full rendered HTML historically at all vs. re-render on demand from stored `render_context` + pinned content versions; if storing, what medium (file/DB/object storage); and how per-recipient historical renders stay accessible from the service layer rather than being opaque static files. **Candidate direction:** rather than persisting full HTML per recipient, store a compact structured reference per recipient per send — `variantId` + resolved `content_record_id`s + position — and reconstruct/re-render on demand when a historical view is needed. This is largely what `SnapshotDB.render_context` (JSON, added commit 431c053) already captures, so this may mostly be a matter of confirming/extending that existing field rather than inventing a new one. **Confirmed gaps to resolve as part of this decision** (Snapshots Q2): `render_context` today captures resolved content/decision data per module but not (1) active overrides — `module_data.*_override` values applied at render time are never recorded, and (2) merge-context (recipient-specific substitution values like firstname) — both explicitly required by ADR-062's "overrides" and the related ADR-005 merge-context concept. **Also needs to answer:** should per-recipient snapshots be persisted proactively at send time at all, or should "what did recipient X actually receive" always be reconstructed on-demand (e.g. by a support/audit service reading the structured log + re-rendering), only when actually requested rather than for every send? Source: [[Interview Prep - Delivery, Providers, Recipients]] → Delivery Q4; [[Interview Prep - Snapshots, Rendering, Email Modules]] → Snapshots Q2.

- 📋 **System-generated "shadow variant" for override-outcome validation — partially resolved 2026-07-05, still needs an ADR for content overrides.** The override-outcome-comparison concept ("70% of overrides didn't outperform the system's pick," `docs/playbook-strategy.md` Decision Log 2026-06-24) needs a real counterfactual to compare against, not a hypothetical "the system would have chosen X" that was never actually sent to anyone. [[ADR-021 — Variants Are Human Created Versions]] frames A/B testing as human-initiated, which is what originally raised the conflict. **Resolved — audience overrides via addition or full acceptance:** confirmed 2026-07-05 — since engagement is tracked per recipient, the manager-added cohort and the system-proposed cohort within one real send are their own natural comparison group; no parallel system-built send needed, no ADR-021 conflict here. **Still open — audience overrides via deletion/exclusion:** people the system proposed but the manager removed never receive anything, so there's no real engagement data for them either — same underlying problem as content overrides, below. Flagged as rare in practice (deletion expected far less often than addition), so deprioritized rather than solved. **Still open — content overrides:** when a manager replaces the system's chosen content, the rejected pick is never sent to anyone, so there's no way to get real engagement data validating "the system's pick would have done just as well" without an actual shadow send or equivalent — this still directly conflicts with [[ADR-021 — Variants Are Human Created Versions]] and needs an ADR amendment or companion ADR before implementation, exactly as originally flagged. See `docs/playbook-strategy.md` Decision Log, 2026-07-05, item 1. Original context: [[Interview Prep - Decision, Overrides, Insight]] → Overrides Q1 (addendum).

- 📋 **Guaranteed placement / sponsored slots — new concept, not yet designed.** Surfaced 2026-07-05 while resolving the personalization-weighting question (Decision Q-adjacent, `recipient_top_score.py` defaults). A content slot that sits *outside* personalization ranking entirely: sales flags an item as "will show no matter what" (usually one-off, last-minute), or a partner contract guarantees minimum placement on an ongoing basis. Deliberately distinct from the override mechanism above — different trigger (a business/contractual guarantee, not a one-off editorial judgment call) and often a different owner (sales/partnerships, not the person building the email). Needs its own design before implementation — not yet scoped as an ADR or a feature. Source: `docs/playbook-strategy.md` Decision Log, 2026-07-05, item 5; `docs/business-interview-baseline.md` §M1.

- 📋 **General data lifecycle / deletion & retention policy — not designed at all yet.** Confirmed during the interview review: the project has no deletion/cleanup story anywhere. Two concrete instances found so far, both symptoms of the same missing policy:
  - **Category/content record deletion vs. archival.** Categories accumulate historical value via preference/engagement scoring (`ContentCategoryAssignmentDB.score`, `RecipientPreferenceDB`) — hard-deleting a category breaks the ability to interpret that historical data (what a past engagement signal even meant). Leaning toward: never hard-delete a category/content record that has any historical usage — **archive** instead. An archived category stops contributing to *new* preference scoring; existing preference influence from it decays naturally over time (same mechanism as the not-yet-built Phase 3C signal decay, [[ADR-112 — Signals Use Time-Based Decay]]) rather than being force-zeroed. Only once a category is fully unused *and* decayed out would actual deletion become safe to consider.
  - **Snapshot lifecycle.** No uniqueness/idempotency key on snapshot creation means unbounded accumulation of near-duplicate snapshots per variant/recipient — consistent with "immutable append-only history" (ADR-061) but with no TTL, retention rule, or delete endpoint at all.

  Needs a dedicated ADR (none exists today) covering deletion/retention policy generally — likely also relevant to other accumulating tables (engagement events, decision resolutions, preference logs) once they're examined the same way. Tightly coupled to the Phase 3C signal layer design already flagged as open in `docs/playbook-strategy.md` §7. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q6 (addendum); [[Interview Prep - Snapshots, Rendering, Email Modules]] → Snapshots Q6.
