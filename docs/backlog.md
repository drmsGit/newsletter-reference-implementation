## Bugs

- üî¥ **[Bug]** Deleting a category or content record that still has relations (parent/child edges, category assignments) should not silently cascade ‚Äî raise a confirmation-style error surfacing what it's connected to ("related to N parent(s) / M child(ren) ‚Äî delete anyway?") so a manager can re-parent/re-assign first if needed. Applies once delete endpoints for content records/categories are built (none exist yet); pairs with the FK/cascade design from Content Q5. Does not cover the harder historical-data question below ‚Äî see the üìã Needs ADR item. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q6.

- üî¥ **[Bug]** `send_send_instance` (`delivery/service.py:160-173`) only calls `db.commit()` once, at the very end of the send loop ‚Äî if `provider.send()` raises mid-batch (e.g. execution 5 of 100), executions 1-4 were actually sent by the provider but never get their `"sent"` status persisted, since the exception aborts the function before the single end-of-loop commit. Keeping "stop the batch on a failure" as the safe default is correct (provider-dependent whether skip-and-continue would even be safe) ‚Äî but whatever succeeded before the stop must still be recorded accurately. Fix: commit incrementally (or otherwise ensure completed executions are persisted) so the DB never claims an email was never sent when it actually was, independent of retry/skip policy. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q3.

- üî¥ **[Bug]** `send_send_instance` never consults `result.success` from `provider.send()` ‚Äî `execution.status` is unconditionally set to `"sent"` regardless of the actual result (`delivery/service.py:168`), and the mock provider always returns `success=True` so this path is never exercised. Distinct from Q3 (that's an exception being raised; this is a returned failure result being ignored). Fix: consult `result.success` when setting status, and log the full result (success/failure/message) for every send attempt, not just failures ‚Äî small observability fix, useful regardless of provider. Defer the harder part (how to react differently to varied real-provider failure modes/retry semantics) until real providers are integrated ‚Äî building that against a mock that only ever succeeds would be guesswork. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q6.

- üî¥ **[Bug]** Nothing stops two concurrent calls from sending the same send instance twice ‚Äî no status guard before the loop runs in `send_send_instance`, no unique constraint or row lock (`delivery/service.py`). Would produce duplicate sends and last-write-wins on `provider_message_id`. Fix: add a status guard (reject/no-op if already `"sending"`/`"sent"`) plus a row lock or unique constraint to make it safe-by-construction, not just application-level. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q5.

- üî¥ **[Bug]** `POST /decision/slots/{id}/execute` (`decision/router.py:24-37` ‚Üí `decision/service.py:9-43`) unconditionally inserts a brand-new `DecisionResolutionDB` row every time it's called, with no check against the recipient's last resolution for that slot ‚Äî confirmed to already happen in practice (re-running for a recurring journey creates another row even when nothing changed). Two fixes, prioritized data-minimization first: (1) non-personalized strategies (`top_score`, `requires_recipient=False`) should only ever have **one** resolution row per slot (`recipient_id=NULL`, already supported by the schema and rendering's read path) ‚Äî never a duplicate per recipient; (2) personalized strategies (`recipient_top_score`) should only write a new row when something that could change the outcome actually changed since the recipient's last resolution (preference update, candidate content change) ‚Äî "no new signal, keep the last recommendation" ‚Äî rather than accumulating an unbounded, ever-growing number of per-person records over time. Storing as little per-recipient data as possible matters more here than the query-performance win, though both improve together. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Decision Q5.

- üî¥ **[Bug]** `send_send_instance` reads one shared, variant-level snapshot and sends identical HTML to every recipient (`delivery/service.py:111-172`) instead of resolving per-recipient HTML ‚Äî even though decision-slot personalization (ADR-083) already exists in the rendering layer and could resolve different content per recipient within the same variant. Note: providers genuinely don't need campaignId/variantId/snapshotId at all (internal bookkeeping, not the ESP's concern), and variantId being identical across recipients within one send is *correct*, not a gap, when personalization stays inside one variant via decision slots ‚Äî it would only legitimately differ per recipient in a true split-delivery/A/B scenario, which is the separate "shadow variant" idea already in Needs ADR. Fix scope, narrowed: resolve/render HTML per recipient (reusing existing per-recipient rendering) before handing each recipient their own final HTML to the provider, instead of reusing one shared snapshot for the whole send. Subject/preheader as dedicated provider fields is tracked separately (Campaigns Q4). Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q2.

- üî¥ **[Bug]** `RecipientTopScoreStrategy`'s tie handling (`recipient_top_score.py:69-76`) is undisclosed, unintentional pseudo-randomness ‚Äî `max()` picks whichever tied candidate the DB happened to return first, with no `ORDER BY` and no mention of the tie in `reason`, undermining ADR-085 explainability and risking a permanent, unexplainable bias toward whichever content has a lower ID. Fix: make tie-breaking an explicit, disclosed random selection among tied candidates, and state it in the `reason` string ("tied at score X between N candidates, selected randomly") ‚Äî honest and auditable rather than hidden. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Decision Q7.

- üî¥ **[Bug]** Rendering has no preview-vs-send resolution mode, and static content modules (`ModuleInstanceDB.content_record_id`) have no version-awareness at all ‚Äî `resolve_content_for_module` always resolves the live, mutable `ContentRecordDB` for static modules regardless of context (`rendering/service.py:196-200,151-188`). Decision-slot content only pins a version if `DecisionResolutionDB.content_version_id` happens to be set ‚Äî nothing ever auto-resolves "the latest version." Net effect: draft edits to a content record are immediately live everywhere, including active/recurring journeys, with no freeze point. Fix: add a send/snapshot resolution mode that queries the latest `ContentVersionDB` for the record and **raises an error ("contains unpublished content") rather than silently falling back to the live record** if no version exists yet ‚Äî so managers can't accidentally send/snapshot draft content. Preview/builder mode keeps resolving against the live record (unchanged). Convenience idea: a "publish all" button to freeze every unpublished record in a variant in one step before sending. Depends on / pairs with the content-schema redesign below (once `content` lives on the record, "create version" is the freeze action this error is checking for). Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q2.

- üî¥ **[Bug]** Both plugin-style registries (`decision/strategies/registry.py:11-29`, `email_modules/registry.py:42-64`) eagerly discover and load every file at import time with no per-item error isolation ‚Äî one broken strategy file (bad import, failing constructor) or one malformed `.json` manifest crashes the entire registry, taking down the whole app at startup. `email_modules/registry.py` already isolates the *missing-counterpart-file* case gracefully (logs a warning, skips just that pair) but not malformed content within a file. Fix: wrap each individual file's discovery/import/instantiation in its own try/except inside both `_discover()` loops ‚Äî log a clear warning naming the broken file, skip just that one, keep the rest of the registry working. Auto-discovery ("drop a file, no code changes") and full graceful degradation aren't in tension ‚Äî this is a standard pattern, not a tradeoff. Apply to both registries in one pass. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Decision Q1 (also affects the not-yet-reviewed email-modules question on the same gap).

- üî¥ **[Bug]** No uniqueness constraint on `(variant_id, position)` for `ModuleInstanceDB` ‚Äî duplicate or gapped positions produce ambiguous render order (undefined secondary sort stability), and there's no reordering endpoint for a builder "insert between" UX (`campaigns/service.py:110-146`). Already observed causing real problems in test mode, not just theoretical. Fix: positions should never be entered manually ‚Äî always auto-generate (e.g. `MAX(position) + 1` within the variant on create). Reordering should swap positions between the two affected modules rather than allow arbitrary/gapped values, so duplicates can never exist. Add a DB-level unique constraint on `(variant_id, position)` as a safety net alongside the application logic. Ranked high ‚Äî confirmed real-world occurrence, not speculative. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Campaigns Q6.

- üî¥ **[Bug]** Two competing category hierarchy mechanisms exist: `CategoryDB.parent_category_id` (leftover from an earlier, wrong implementation) and `CategoryRelationDB` (parent_category_id/child_category_id/relation_type edge table, already what the graph view at `frontend/router.py:1608-1621` actually uses). Fix: drop `CategoryDB.parent_category_id` entirely and consolidate on `CategoryRelationDB` as the single mechanism. Deliberately keep it fully flexible, no new restrictions: a child may have multiple parent relations, and parent-level (main) categories may relate to each other or to children freely ‚Äî needed for real-world multi-parent taxonomies and for AI-discovered relations later, and adding artificial constraints now isn't necessary. Do add a cycle guard (reject a relation that would create a direct or transitive parent_child loop, e.g. A‚ÜíB and B‚ÜíA) since that's a genuine logical contradiction, not a business-flexibility case. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q5.

- üî¥ **[Bug]** `DecisionResolutionDB.score` is stored as `Integer`, so `score=int(result.score)` (`decision/service.py:42`) silently truncates the weighted float score `recipient_top_score` computes ‚Äî two candidates differing only in fractional score (12.4 vs 12.9) both round to 12, losing tie-breaking/explainability granularity relevant to ADR-085. Fix: change the column to a proper decimal/float type (e.g. `Numeric`) and drop the `int()` cast. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Decision Q6.

- üî¥ **[Bug]** `ModuleInstanceDB` has both `content_record_id` and `decision_slot_id` as independent nullable FKs with nothing enforcing mutual exclusivity ‚Äî if both are set, rendering silently prefers `content_record_id` and ignores the decision slot (`rendering/service.py:196-205`); if neither is set, the module renders nothing. Considered replacing with a generic `type`+`id` polymorphic column but rejected ‚Äî that's a known anti-pattern that loses real FK referential integrity and forces conditional/manual joins, for a type-count (2) too small to justify it. Fix: keep the two nullable FK columns, add a mutual-exclusivity constraint (DB `CHECK` or Pydantic/service-layer validator) enforcing exactly one is set. `campaigns/db_models.py:37-53`, `models.py:50-55`. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Campaigns Q2.

- üî¥ **[Bug]** `recipient_id` means two different things across modules ‚Äî `recipients`/`decision`/`insight` use the internal integer PK, while `delivery`/`providers` (`DeliveryExecutionDB.recipient_id`) use the string `external_id`, with no FK between them. This forces indirect lookups (e.g. `insight/service.py:105-118` has to query `RecipientDB WHERE external_id = ...` just to translate a delivery execution's recipient into something joinable), and each such lookup is an extra failure mode (mismatched/stale external_id ‚Üí `ValueError`) that a real FK wouldn't have. Confirmed this doesn't add any new CRM integration burden ‚Äî per [[ADR-126 ‚Äî Maintain Local Recipient Projection]], delivery execution tracking already requires the local `RecipientDB` projection to exist and be synced regardless of which of its two ID columns gets referenced; switching to the internal PK is purely internal plumbing, not a new sync dependency. Fix: make `DeliveryExecutionDB.recipient_id` (and any other cross-module reference doing the same thing) a direct FK to `RecipientDB.id`, consistent with [[ADR-054 ‚Äî Use Internal Recipient Identifiers]]. Good time to do this ‚Äî data model is still cheap to reshape. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Insight Q5 (also flagged as a cross-cutting finding in [[MOC - Interview Prep Baseline]], affects the not-yet-reviewed Delivery/Recipients cluster too).

- üî¥ **[Bug]** `ContentCategoryAssignmentDB.score` has no validation at write or read time ‚Äî can be zero (degenerate/meaningless; "no relevance" should mean no assignment row, not a stored zero) or exceed the 0-10 range, and `RecipientPreferenceDB.new_score` has no ceiling so it can grow unbounded, compounding into `RecipientTopScoreStrategy`'s ranking (`recipients/service.py:135,170`). Fix now: reject a score of exactly zero at write time. The 0-10 range itself is a POC-only convention, not a hard rule ‚Äî fold into the existing config-layer future work (Insight Q2) so a deployment can set its own limits rather than hardcoding 0-10. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Insight Q4.

- üî¥ **[Bug]** The dedupe key in `apply_event_to_preferences` ‚Äî `(recipient_id, category_id, reason=event_type, delivery_execution_id)` ‚Äî doesn't identify the specific event, so two distinct legitimate engagements (e.g. clicks on two different links, same email/category/execution) get conflated: the second is silently dropped as a "duplicate" and never scored (`recipients/service.py:137-150`). Every real event must still be saved (already true for the raw `EngagementEventDB` row) ‚Äî deduplication should only decide *whether/how much score* a given event contributes, not whether the event itself is recorded. Fix: include something that uniquely identifies the specific engagement (e.g. `event_id`) in the dedupe key, so distinct events each get their own scoring decision while true duplicate deliveries of the *same* event still dedupe correctly. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Insight Q3.

- üî¥ **[Bug]** `OverrideEventDB.override_type` is unconstrained free text ‚Äî only `"content" | "segment"` documented in a code comment, not validated anywhere. Fix: make it a proper enum. Small, standalone fix, separate from the reset-mechanism question logged under the override-strategy redesign Feature. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q5.

- üî¥ **[Bug]** `record_outcome_delta` (`overrides/service.py:32-39`) replaces `outcome_delta` wholesale on each PATCH instead of merging keys ‚Äî outcome data that arrives incrementally (e.g. open-rate today, click-rate a week later) has the second update silently overwrite the first entirely. `test_overrides.py:130-138` only checks non-`outcome_delta` fields survive; the merge case is untested and would currently fail. Fix: merge incoming keys into the existing `outcome_delta` dict instead of replacing it. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q3.

- üî¥ **[Bug]** `record_outcome_delta` (`overrides/service.py:32-39`) is a read-modify-write with no lock ‚Äî concurrent PATCH calls computing outcome deltas for the same `override_id` (e.g. an open-rate job and a click-rate job overlapping) have last-write-wins semantics, silently dropping whichever update committed first. Compounds the Q3 merge bug above. Fix mechanism not yet decided ‚Äî either optimistic locking (a `version`/`updated_at` check-and-increment on write) or a DB-level row lock (`SELECT ... FOR UPDATE`); needs a decision before implementing. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q4.

- üî¥ **[Bug]** `OverrideEventCreate` (`overrides/models.py:6-14`) has no validation that `system_content_record_id` and `override_content_record_id` actually exist or differ from each other ‚Äî FK enforcement depends on whether `PRAGMA foreign_keys` happens to be on for the dev DB, so a bad ID silently creates an orphaned audit row. Same systemic silent-orphan pattern as Campaigns Q5 below. Fix: validate both referenced IDs exist (and aren't identical) before insert. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q2.

- üî¥ **[Bug]** `create_decision_resolution` (`campaigns/service.py:235-257`) accepts arbitrary `content_record_id`/`content_version_id`/`recipient_id` with no existence validation before insert ‚Äî depending on whether the DB engine enforces FK constraints (often off by default in SQLite), a bad ID silently creates an orphaned resolution row that only breaks later, at rendering's join-based lookup. Fix: no orphan row should ever be silently accepted ‚Äî validate referenced IDs exist before insert (or ensure FK enforcement is actually on, e.g. `PRAGMA foreign_keys=ON` for SQLite) and return a clear error otherwise. Worth checking whether other `create_*` functions across the codebase have the same silent-orphan gap once this is fixed, since it's likely a systemic pattern, not unique to this one function. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Campaigns Q5.

- üî¥ **[Bug]** `add_member`/`bulk_add_members` (`audience/service.py:54-69,72-85,125-134`) do a SELECT-then-INSERT with no unique constraint on `(group_id, recipient_id)` ‚Äî a TOCTOU race under concurrent calls can produce duplicate membership rows. Fix: add a DB-level unique constraint on `(group_id, recipient_id)` on `AudienceGroupMemberDB`, making it safe-by-construction instead of relying on the application-level existence check. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Audience Q3.

- üî¥ **[Bug]** `AudienceGroupDB.name` uniqueness is case-sensitive and unhandled ‚Äî "Newsletter VIPs" and "newsletter vips" succeed as two distinct groups, and a genuine duplicate name raises an unhandled 500 instead of a clean error (`router.py:16-18`, `service.py:15-33`). Fix: enforce case-insensitive uniqueness and catch `IntegrityError` to return a clean 409 with a clear message. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Audience Q5.

- üî¥ **[Bug]** `get_content_record_by_id` (`content/router.py:43-49`) hand-rolls its own `ContentRecordDB` ‚Üí `ContentRecord` mapping (plus an unnecessary local re-import of `ContentRecord`) instead of using a shared mapper ‚Äî duplicates the same mapping already in `list_content_records`/`update_content_record` (`service.py:17-25,28-39`). Fix: extract a `to_content_record(record) -> ContentRecord` helper in `service.py` (mirroring the existing `to_content_version`/`to_category_relation` pattern) and use it in all three places. Small cleanup. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q4.

- üî¥ **[Bug]** `create_campaign` (`campaigns/service.py:33-61`) auto-creates the initial `VariantDB` via two independent `db.commit()` calls rather than one transaction ‚Äî a failure between them (e.g. DB connection drop) leaves a persisted campaign with zero variants, violating the "a campaign always has a variant" invariant. Fix: wrap both inserts in a single transaction (flush + one commit at the end) so it's atomic. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Campaigns Q1.

- üî¥ **[Bug]** `assign_category_to_content` (`content/service.py:241-257`) has no duplicate check, unlike `audience`'s `add_member` ‚Äî the same content/category pair can be assigned more than once, creating duplicate rows and duplicate entries in `list_categories_for_content` output. Fix: reject (or upsert) on duplicate `(content_id, category_id)` pairs ‚Äî no duplicates allowed. Cosmetic/data-hygiene only (no correctness risk beyond a repeated list item), lowest priority of the bugs logged so far. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q7.

## Features

- üî¥ **[Feature]** Systemic gap: the project needs a general pattern for concurrency guards and validation checks across write endpoints, not just ad hoc per-issue fixes. Two concrete instances already found independently ‚Äî the Overrides Q4 outcome-delta race (`record_outcome_delta`, read-modify-write with no lock) and the Delivery Q5 double-send race ‚Äî are both symptoms of the same missing systemic pattern, including the general case of **two managers editing the same object at the same time** (optimistic locking / edit-conflict detection), which isn't handled anywhere in the project yet. Needs a decision on a standard mechanism (e.g. a `version`/`updated_at` optimistic-lock convention applied consistently) rather than solving each occurrence differently. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q5 (addendum); also ties to [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q4.

- üî¥ **[Feature]** New idea, not yet designed, provider-capability-dependent (ADR-101): proactive partial-failure reconciliation for batch sends ‚Äî when a mid-batch send failure is detected, have the system automatically check the provider's own send log/events for the affected send and surface "these N recipients already received this send ‚Äî exclude them from a retry?" instead of requiring manual lookup via the provider's dashboard. Lowest priority ‚Äî parked idea, explicitly deferred as an IT/ops concern for now rather than something to automate immediately. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q3 (addendum).

- üî¥ **[Feature]** Build the Signal layer (Phase 3C, [[ADR-110 ‚Äî Insight Layer Transforms Events Into Signals]], [[ADR-112 ‚Äî Signals Use Time-Based Decay]], [[ADR-113 ‚Äî Separate Operational and Historical Signals]]) ‚Äî replace `apply_event_to_preferences`'s direct writes to mutable `RecipientPreferenceDB.score` with a proper Signal abstraction supporting decay and operational/historical separation, which is structurally impossible today without a schema change. Design before building (signal types, decay model, storage) per the existing Open Question in `docs/playbook-strategy.md` ¬ß7 ‚Äî don't re-derive scope here, that's the design doc. **This is a foundational dependency for two other backlog items already waiting on it:** the category/content archive-vs-decay mechanism (Needs ADR, Content Q6 addendum) and the AI-recommendation-on-taxonomy feature (Content Q5 addendum). Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Insight Q1.

- üî¥ **[Feature]** Negative category-affinity scores ("clicking X signals *reduced* interest in category Y") ‚Äî decided direction: this is a **system proposal only**, surfaced via the future Signal layer/AI analysis (it needs real behavioral evidence to get the sign/magnitude right), with the existing override mechanism as the channel for a manager to accept, reject, or manually adjust it. Explicitly **not** a general "just type in a negative score" input ‚Äî no freeform manual negative-scoring capability. Depends on the Signal layer (Insight Q1) and the override-strategy redesign (Campaigns Q3/Overrides Q1) both landing first. **Confirmed connection:** the category impact graph view already has a working `avg_delta < 0` display branch (`frontend/router.py:1765-1767`, fed by `AVG(PreferenceUpdateLogDB.delta)`) ‚Äî today it's only reachable via the Q4 validation bug (an accidentally-entered negative assignment score), not any legitimate signal. This feature is what would make that existing UI branch meaningfully fire on purpose, once the Q4 zero/range validation closes off the accidental path. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Insight Q4 (addendum).

- üî¥ **[Feature]** New idea, not yet designed: a "most recipients matched" tiebreak variant for personalized decision strategies ‚Äî when candidates tie in `recipient_top_score`, prefer the one that also matches the most *other* recipients (a global popularity/social-proof signal), instead of (or as an option alongside) the explicit random tiebreak. This is a bigger design question than a simple tiebreaker ‚Äî it injects a cross-recipient signal into what's currently a purely individual-preference-driven strategy, and would need its own query/strategy design (possibly a distinct strategy variant, e.g. `recipient_top_score_popularity_tiebreak`, rather than baked into the existing one). Lowest priority ‚Äî parked as an idea, not scoped. Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Decision Q7 (alternative considered, kept separate).

- üî¥ **[Feature]** Build one standardized pagination mechanism (shared query-param dependency + response envelope, e.g. `{items, total, limit, offset}`) and apply it across every data-heavy list endpoint: content records, audience groups (`find_by_criteria`/`list_members`), content versions, decision slots/resolutions, engagement events, recipients. No pagination pattern exists today ‚Äî only scattered hardcoded `.limit(20)`/`.limit(50)` calls in `frontend/router.py` with no offset/page param, no consistent shape. One offset/limit-based approach is sufficient for all of these at this project's realistic scale (none will hit true big-data volumes needing cursor/keyset pagination). Ranked reasonably high: even in the POC, presenting to people needs realistic test data volumes (recipients, events especially), and a positive UX matters in the POC too, not just "the real version." Supersedes the earlier "keep in poc" call on content-record pagination ‚Äî folded in here. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Audience Q6 (supersedes Content Q3).

- üî¥ **[Feature]** Decision strategies should declare their expected `candidate_filter`/`strategy_config` shape (similar to how email module templates already declare a `variables` manifest, `email_modules/registry.py`), and the UI/API should enforce it: once a manager picks a `decision_strategy` on a decision slot, the config/filter **structure** locks to that strategy's declared shape ‚Äî only values are editable, not structure ‚Äî and switching strategies re-derives the structure for the new one. Fixes `update_decision_slot` (`campaigns/service.py:149-164`) currently accepting any shape for any strategy, only failing (or previously crashing) at resolution time. Principle: anything touching human/manager input should be as intuitive and mistake-proof as possible. Ties into the existing decision-strategy plugin contract hardening (commit cf4f745, Phase 1A). Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Campaigns Q7.

- üî¥ **[Feature]** Add a bulk-remove-by-criteria function mirroring `bulk_add_members` (`audience/service.py:125-134`) ‚Äî currently only bulk-add exists, no bulk-remove counterpart. Should track which criteria added each member (e.g. a `source_criteria` column on `AudienceGroupMemberDB`) so that removing one criterion only removes recipients who aren't still validly in the group via a *different* active criterion, rather than blanket-removing anyone matching the criteria being un-applied. Needs some design thinking, not fully spec'd yet ‚Äî not necessarily ADR-level, but more than a one-line fix. Related: the criteria-preview (`find_by_criteria`, `service.py:112-118`) currently has no `.distinct()`, so a recipient matching via multiple preference rows for the same category appears twice ‚Äî rather than dedup that away, surface *which* criteria matched each recipient in the preview too (reusing the same source-criteria tracking), so a manager can see why someone's in a segment before removing a criterion. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Audience Q2, Q4.

- üî¥ **[Feature]** Design one coherent, scalable override strategy and rebuild the two known-inconsistent override mechanisms onto it: (1) `ModuleInstanceDB.module_data` mixes structural config with content overrides (`headline_override`, etc.) in one untyped JSON blob, no schema boundary, no way to query "which modules have an active override" (`campaigns/db_models.py:45`); (2) the newer `OverrideEventDB` audit table is completely disconnected from rendering, which still reads the old `module_data.*_override` blob (flagged as the top cross-cutting finding in [[MOC - Interview Prep Baseline]]). ## Needs ADR (blocked on design decision)

_Not prioritized alongside üî¥ To Do items ‚Äî these need a decision recorded as an ADR before they're actionable. Move to Bugs/Features once resolved._

- üìã **Finalize the rendered-HTML/snapshot storage strategy ‚Äî now more urgent given per-recipient personalization.** Currently writes flat HTML files (`storage/snapshots/`), with `SnapshotDB.html_storage_type` as an unused extension point for object storage (per project notes: this was already an open decision). Commit 380ebe1 worked around a related seed-data bug (bad `html_location` paths) rather than resolving the underlying strategy, and left a side-effect NULL in `PreferenceUpdateLogDB.event_id` (a `nullable=False` column). The Delivery Q2 fix (per-recipient HTML resolution instead of one shared variant-level snapshot) raises the stakes: instead of one rendered artifact per variant, this could mean one per recipient per send ‚Äî making "do we even need to persist full HTML files at all" a much more pressing question than before. Needs a decision on: whether to store full rendered HTML historically at all vs. re-render on demand from stored `render_context` + pinned content versions; if storing, what medium (file/DB/object storage); and how per-recipient historical renders stay accessible from the service layer rather than being opaque static files. **Candidate direction:** rather than persisting full HTML per recipient, store a compact structured reference per recipient per send ‚Äî `variantId` + resolved `content_record_id`s + position ‚Äî and reconstruct/re-render on demand when a historical view is needed. This is largely what `SnapshotDB.render_context` (JSON, added commit 431c053) already captures, so this may mostly be a matter of confirming/extending that existing field rather than inventing a new one. Source: [[Interview Prep - Delivery, Providers, Recipients]] ‚Üí Delivery Q4.

- üìã **System-generated "shadow variant" for override-outcome validation ‚Äî conflicts with an existing ADR.** The override-outcome-comparison concept ("70% of overrides didn't outperform the system's pick," `docs/playbook-strategy.md` Decision Log 2026-06-24) needs a real counterfactual to compare against, not a hypothetical "the system would have chosen X" that was never actually sent to anyone. [[ADR-021 ‚Äî Variants Are Human Created Versions]] frames A/B testing as human-initiated ("A/B test versions are variants... delivery logic belongs outside the Builder"), but what's actually needed here is different: an ongoing, automatic system check that runs whenever an override exists ‚Äî not a deliberate human-initiated test. Proposed direction: the system builds its own "shadow variant" directly before send whenever overrides are present, so a manager can see what the system would have actually built/sent and to how many people, with a "cancel system send" option. This **directly contradicts ADR-021's core decision that a variant is always human-created** ‚Äî needs an ADR amendment or a companion ADR distinguishing "human-initiated A/B variant" from "system-generated shadow variant for override validation" before implementation. Also still depends on real split-delivery logic existing (see the Delivery cluster's variant-delivery gap, not yet reviewed). Source: [[Interview Prep - Decision, Overrides, Insight]] ‚Üí Overrides Q1 (addendum).

- üìã **Category (and content record) deletion vs. archival policy.** Categories accumulate historical value via preference/engagement scoring (`ContentCategoryAssignmentDB.score`, `RecipientPreferenceDB`) ‚Äî hard-deleting a category breaks the ability to interpret that historical data (what a past engagement signal even meant). Leaning toward: never hard-delete a category/content record that has any historical usage ‚Äî **archive** instead. An archived category stops contributing to *new* preference scoring; existing preference influence from it decays naturally over time (same mechanism as the not-yet-built Phase 3C signal decay, [[ADR-112 ‚Äî Signals Use Time-Based Decay]]) rather than being force-zeroed. Only once a category is fully unused *and* decayed out would actual deletion become safe to consider. Needs a dedicated ADR (none exists today covering archive-vs-delete policy) before implementation ‚Äî this is tightly coupled to the Phase 3C signal layer design already flagged as open in `docs/playbook-strategy.md` ¬ß7. Source: [[Interview Prep - Content, Campaigns, Audience]] ‚Üí Content Q6 (addendum).
