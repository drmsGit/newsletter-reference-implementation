# Backlog — Action Items from Interview-Prep Reviews

Granular, prioritized queue of concrete fixes/features decided while working through [[MOC - Interview Prep Baseline]] via the `/interview-review` command. Distinct from the phase-level roadmap in `docs/playbook-strategy.md` §6 (Phase 1-4, strategic/product sequencing) — this is the tactical "what do we act on, in what order" list.

Order within each section is priority order: **top = do next.** New items are inserted at the position matching their priority, not appended to the bottom. Bugs outrank features of similar importance.

## Bugs

- 🔴 **[Bug]** Deleting a category or content record that still has relations (parent/child edges, category assignments) should not silently cascade — raise a confirmation-style error surfacing what it's connected to ("related to N parent(s) / M child(ren) — delete anyway?") so a manager can re-parent/re-assign first if needed. Applies once delete endpoints for content records/categories are built (none exist yet); pairs with the FK/cascade design from Content Q5. Does not cover the harder historical-data question below — see the 📋 Needs ADR item. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q6.

- 🔴 **[Bug]** Rendering has no preview-vs-send resolution mode, and static content modules (`ModuleInstanceDB.content_record_id`) have no version-awareness at all — `resolve_content_for_module` always resolves the live, mutable `ContentRecordDB` for static modules regardless of context (`rendering/service.py:196-200,151-188`). Decision-slot content only pins a version if `DecisionResolutionDB.content_version_id` happens to be set — nothing ever auto-resolves "the latest version." Net effect: draft edits to a content record are immediately live everywhere, including active/recurring journeys, with no freeze point. Fix: add a send/snapshot resolution mode that queries the latest `ContentVersionDB` for the record and **raises an error ("contains unpublished content") rather than silently falling back to the live record** if no version exists yet — so managers can't accidentally send/snapshot draft content. Preview/builder mode keeps resolving against the live record (unchanged). Convenience idea: a "publish all" button to freeze every unpublished record in a variant in one step before sending. Depends on / pairs with the content-schema redesign below (once `content` lives on the record, "create version" is the freeze action this error is checking for). Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q2.

- 🔴 **[Bug]** No uniqueness constraint on `(variant_id, position)` for `ModuleInstanceDB` — duplicate or gapped positions produce ambiguous render order (undefined secondary sort stability), and there's no reordering endpoint for a builder "insert between" UX (`campaigns/service.py:110-146`). Already observed causing real problems in test mode, not just theoretical. Fix: positions should never be entered manually — always auto-generate (e.g. `MAX(position) + 1` within the variant on create). Reordering should swap positions between the two affected modules rather than allow arbitrary/gapped values, so duplicates can never exist. Add a DB-level unique constraint on `(variant_id, position)` as a safety net alongside the application logic. Ranked high — confirmed real-world occurrence, not speculative. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q6.

- 🔴 **[Bug]** Two competing category hierarchy mechanisms exist: `CategoryDB.parent_category_id` (leftover from an earlier, wrong implementation) and `CategoryRelationDB` (parent_category_id/child_category_id/relation_type edge table, already what the graph view at `frontend/router.py:1608-1621` actually uses). Fix: drop `CategoryDB.parent_category_id` entirely and consolidate on `CategoryRelationDB` as the single mechanism. Deliberately keep it fully flexible, no new restrictions: a child may have multiple parent relations, and parent-level (main) categories may relate to each other or to children freely — needed for real-world multi-parent taxonomies and for AI-discovered relations later, and adding artificial constraints now isn't necessary. Do add a cycle guard (reject a relation that would create a direct or transitive parent_child loop, e.g. A→B and B→A) since that's a genuine logical contradiction, not a business-flexibility case. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q5.

- 🔴 **[Bug]** `ModuleInstanceDB` has both `content_record_id` and `decision_slot_id` as independent nullable FKs with nothing enforcing mutual exclusivity — if both are set, rendering silently prefers `content_record_id` and ignores the decision slot (`rendering/service.py:196-205`); if neither is set, the module renders nothing. Considered replacing with a generic `type`+`id` polymorphic column but rejected — that's a known anti-pattern that loses real FK referential integrity and forces conditional/manual joins, for a type-count (2) too small to justify it. Fix: keep the two nullable FK columns, add a mutual-exclusivity constraint (DB `CHECK` or Pydantic/service-layer validator) enforcing exactly one is set. `campaigns/db_models.py:37-53`, `models.py:50-55`. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q2.

- 🔴 **[Bug]** `create_decision_resolution` (`campaigns/service.py:235-257`) accepts arbitrary `content_record_id`/`content_version_id`/`recipient_id` with no existence validation before insert — depending on whether the DB engine enforces FK constraints (often off by default in SQLite), a bad ID silently creates an orphaned resolution row that only breaks later, at rendering's join-based lookup. Fix: no orphan row should ever be silently accepted — validate referenced IDs exist before insert (or ensure FK enforcement is actually on, e.g. `PRAGMA foreign_keys=ON` for SQLite) and return a clear error otherwise. Worth checking whether other `create_*` functions across the codebase have the same silent-orphan gap once this is fixed, since it's likely a systemic pattern, not unique to this one function. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q5.

- 🔴 **[Bug]** `add_member`/`bulk_add_members` (`audience/service.py:54-69,72-85,125-134`) do a SELECT-then-INSERT with no unique constraint on `(group_id, recipient_id)` — a TOCTOU race under concurrent calls can produce duplicate membership rows. Fix: add a DB-level unique constraint on `(group_id, recipient_id)` on `AudienceGroupMemberDB`, making it safe-by-construction instead of relying on the application-level existence check. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q3.

- 🔴 **[Bug]** `AudienceGroupDB.name` uniqueness is case-sensitive and unhandled — "Newsletter VIPs" and "newsletter vips" succeed as two distinct groups, and a genuine duplicate name raises an unhandled 500 instead of a clean error (`router.py:16-18`, `service.py:15-33`). Fix: enforce case-insensitive uniqueness and catch `IntegrityError` to return a clean 409 with a clear message. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q5.

- 🔴 **[Bug]** `get_content_record_by_id` (`content/router.py:43-49`) hand-rolls its own `ContentRecordDB` → `ContentRecord` mapping (plus an unnecessary local re-import of `ContentRecord`) instead of using a shared mapper — duplicates the same mapping already in `list_content_records`/`update_content_record` (`service.py:17-25,28-39`). Fix: extract a `to_content_record(record) -> ContentRecord` helper in `service.py` (mirroring the existing `to_content_version`/`to_category_relation` pattern) and use it in all three places. Small cleanup. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q4.

- 🔴 **[Bug]** `create_campaign` (`campaigns/service.py:33-61`) auto-creates the initial `VariantDB` via two independent `db.commit()` calls rather than one transaction — a failure between them (e.g. DB connection drop) leaves a persisted campaign with zero variants, violating the "a campaign always has a variant" invariant. Fix: wrap both inserts in a single transaction (flush + one commit at the end) so it's atomic. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q1.

- 🔴 **[Bug]** `assign_category_to_content` (`content/service.py:241-257`) has no duplicate check, unlike `audience`'s `add_member` — the same content/category pair can be assigned more than once, creating duplicate rows and duplicate entries in `list_categories_for_content` output. Fix: reject (or upsert) on duplicate `(content_id, category_id)` pairs — no duplicates allowed. Cosmetic/data-hygiene only (no correctness risk beyond a repeated list item), lowest priority of the bugs logged so far. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q7.

## Features

- 🔴 **[Feature]** Build one standardized pagination mechanism (shared query-param dependency + response envelope, e.g. `{items, total, limit, offset}`) and apply it across every data-heavy list endpoint: content records, audience groups (`find_by_criteria`/`list_members`), content versions, decision slots/resolutions, engagement events, recipients. No pagination pattern exists today — only scattered hardcoded `.limit(20)`/`.limit(50)` calls in `frontend/router.py` with no offset/page param, no consistent shape. One offset/limit-based approach is sufficient for all of these at this project's realistic scale (none will hit true big-data volumes needing cursor/keyset pagination). Ranked reasonably high: even in the POC, presenting to people needs realistic test data volumes (recipients, events especially), and a positive UX matters in the POC too, not just "the real version." Supersedes the earlier "keep in poc" call on content-record pagination — folded in here. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q6 (supersedes Content Q3).

- 🔴 **[Feature]** Decision strategies should declare their expected `candidate_filter`/`strategy_config` shape (similar to how email module templates already declare a `variables` manifest, `email_modules/registry.py`), and the UI/API should enforce it: once a manager picks a `decision_strategy` on a decision slot, the config/filter **structure** locks to that strategy's declared shape — only values are editable, not structure — and switching strategies re-derives the structure for the new one. Fixes `update_decision_slot` (`campaigns/service.py:149-164`) currently accepting any shape for any strategy, only failing (or previously crashing) at resolution time. Principle: anything touching human/manager input should be as intuitive and mistake-proof as possible. Ties into the existing decision-strategy plugin contract hardening (commit cf4f745, Phase 1A). Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q7.

- 🔴 **[Feature]** Add a bulk-remove-by-criteria function mirroring `bulk_add_members` (`audience/service.py:125-134`) — currently only bulk-add exists, no bulk-remove counterpart. Should track which criteria added each member (e.g. a `source_criteria` column on `AudienceGroupMemberDB`) so that removing one criterion only removes recipients who aren't still validly in the group via a *different* active criterion, rather than blanket-removing anyone matching the criteria being un-applied. Needs some design thinking, not fully spec'd yet — not necessarily ADR-level, but more than a one-line fix. Related: the criteria-preview (`find_by_criteria`, `service.py:112-118`) currently has no `.distinct()`, so a recipient matching via multiple preference rows for the same category appears twice — rather than dedup that away, surface *which* criteria matched each recipient in the preview too (reusing the same source-criteria tracking), so a manager can see why someone's in a segment before removing a criterion. Source: [[Interview Prep - Content, Campaigns, Audience]] → Audience Q2, Q4.

- 🔴 **[Feature]** Design one coherent, scalable override strategy and rebuild the two known-inconsistent override mechanisms onto it: (1) `ModuleInstanceDB.module_data` mixes structural config with content overrides (`headline_override`, etc.) in one untyped JSON blob, no schema boundary, no way to query "which modules have an active override" (`campaigns/db_models.py:45`); (2) the newer `OverrideEventDB` audit table is completely disconnected from rendering, which still reads the old `module_data.*_override` blob (flagged as the top cross-cutting finding in [[MOC - Interview Prep Baseline]]). Go through [[ADR-040 — Introduce Override Layer]] and [[ADR-041 — Override Precedence]] and design a single mechanism that's explicit, queryable, and consistent — the current split reads as unintentional/broken to anyone unfamiliar with the project's history, even though the underlying reasoning was sound at each step. Resolve alongside the Overrides-cluster question on `OverrideEventDB` when that file is reviewed. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q3.

- 🔴 **[Feature]** Add `subject`/`preheader` fields to `CampaignDB`/`VariantDB` (currently missing — `send_instance.name` is reused as the subject at send time, `delivery/service.py:164`, conflating an internal label with the actual email subject line; same gap independently flagged in the Delivery cluster's review of `send_send_instance`). Part of a broader "campaigns/content need more first-class fields (settings, etc.)" effort not yet done — not an oversight in this one spot, just not reached yet. Source: [[Interview Prep - Content, Campaigns, Audience]] → Campaigns Q4.

- 🔴 **[Feature]** Redesign content schema: move full CMS content fields onto `ContentRecordDB` as a `content` JSON column (same shape as `ContentVersionDB.content`, e.g. `headline_medium`/`body_medium`), so record creation/editing carries the real renderable content directly instead of requiring a separate "create version" step to enter it. Keep `title` (+ short `description`) as metadata columns for catalog listing/filtering. "Create version" becomes a freeze/snapshot action (stand-in for future publish) that copies the record's current `content` into an immutable `ContentVersionDB` row for audit/rollback (ADR-128) — not a "now type your content" step. Also resolves the root cause of the hardcoded `headline_medium`/`body_medium` alias patch in `rendering/service.py` (record and version will finally share one content shape). Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q1.

- 🔴 **[Feature]** AI/system recommendation layer on top of the (human-governed, per [[ADR-080 — Human-governed Taxonomy Before AI Selection]]) category taxonomy — e.g. "system suggests promoting this to a main category" or "system suggests this relates to category X" based on usage frequency/scoring. Reuses the same propose-don't-impose trust-building pattern already designed for content overrides (`docs/playbook-strategy.md` Decision Log, 2026-06-24) rather than making hierarchy itself emergent/scored (rejected — see Content Q5 addendum). Depends on the Phase 3C signal layer (usage/frequency signals don't exist yet) — future work, not current priority. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q5 (addendum).

## Needs ADR (blocked on design decision)

_Not prioritized alongside 🔴 To Do items — these need a decision recorded as an ADR before they're actionable. Move to Bugs/Features once resolved._

- 📋 **Category (and content record) deletion vs. archival policy.** Categories accumulate historical value via preference/engagement scoring (`ContentCategoryAssignmentDB.score`, `RecipientPreferenceDB`) — hard-deleting a category breaks the ability to interpret that historical data (what a past engagement signal even meant). Leaning toward: never hard-delete a category/content record that has any historical usage — **archive** instead. An archived category stops contributing to *new* preference scoring; existing preference influence from it decays naturally over time (same mechanism as the not-yet-built Phase 3C signal decay, [[ADR-112 — Signals Use Time-Based Decay]]) rather than being force-zeroed. Only once a category is fully unused *and* decayed out would actual deletion become safe to consider. Needs a dedicated ADR (none exists today covering archive-vs-delete policy) before implementation — this is tightly coupled to the Phase 3C signal layer design already flagged as open in `docs/playbook-strategy.md` §7. Source: [[Interview Prep - Content, Campaigns, Audience]] → Content Q6 (addendum).
